# 贡献指南

感谢你关注 `Aster Archive`。

这个项目是一个个人维护的 Web 闯关游戏仓库。它既包含站点代码，也包含关卡内容、题面设计、说明文档，以及一些带有明显作者风格的实现选择。因此，这里的协作方式和普通工具库或通用框架项目并不完全一样。

在提交改动之前，建议先花一点时间理解这个项目的目标。它并不是单纯追求“更标准”“更工程化”或“更像常规产品”，而是希望把网页、协议、前后端细节本身变成玩法的一部分。很多看起来不够常规的地方，可能正是关卡设计的一部分。

## 欢迎什么样的贡献

欢迎以下类型的贡献：

- 修复明确的实现错误
- 改进会影响正常游玩的页面问题
- 修复后端 API 的非预期行为
- 补充或改进测试
- 改善文档可读性
- 优化部署、维护或项目结构说明
- 提出新的关卡思路、玩法方向或题面建议
- 改进不破坏设计意图的可访问性与基础体验问题

如果你不确定某个想法是否合适，先开 Discussion 往往比直接提交更稳妥。

## 不建议直接提交的改动

以下改动不建议未经讨论直接发 PR：

- 大幅重写已有关卡
- 直接改变关卡解法路径或核心考点
- 把项目整体改造成另一种技术栈
- 以“更标准”或“更现代”为理由抹平原本有意保留的设计痕迹
- 提前公开会明显破坏游玩体验的隐藏内容
- 只基于个人代码风格偏好做的大规模格式化
- 将页面统一改造成与现有风格不一致的模板化外观

这个项目允许主观性，也接受不完全“产品化”的部分。贡献的重点应当是让它更稳、更清楚、更好玩，而不是把它改造成另一个项目。

## 开始之前

如果你准备做的事情比较小，比如修一个明显 bug、补一段文档、改一个错字，通常可以直接提交。

如果你准备做的事情会影响以下内容，建议先开 Issue 或 Discussion：

- 关卡玩法
- 页面文案
- 提示设计
- API 行为
- 部署方式
- 项目结构
- 许可证、文档策略或公开内容边界

提前对齐方向，通常比做完再解释更有效。

## 贡献时请尽量遵守的原则

### 保留关卡设计意图

很多页面、接口、响应方式和目录结构并不只是“实现细节”，同时也是谜题的一部分。改动前请先判断你看到的是 bug，还是故意留下的题面元素。

### 小步修改

优先提交小而清晰的改动。一个 PR 最好只解决一个问题，或者只处理一类内容。这样更容易审阅，也更不容易误伤已有关卡。

### 不要随意泄露答案

如果改动涉及关卡答案、隐藏路径、关键提示、后端真实逻辑，请控制公开信息的粒度。公开讨论时尽量避免直接把解答写进标题、提交说明或 PR 描述里。

### 文档要服务于维护

如果你的改动会影响后端行为、请求路径、特殊协议玩法、部署方式或测试方法，请同步更新相关文档。项目已经有独立的架构说明和解法文档，README 只负责概要介绍，不要把所有内容都堆回 README。

### 尊重现有风格

这个项目允许页面之间风格不同，也允许实现上有一定手工感。贡献时不需要把一切都抹平到完全统一，但应该避免引入明显破坏现有节奏的改动。

## 仓库结构速览

如果你是第一次看这个仓库，可以先这样理解：

- `public/`：首页、起始页、各关卡页面，以及脚本、字体、样式等静态资源
- `src/`：Node/Express 服务入口和需要后端能力的关卡实现
- `test/`：关键后端关卡的测试
- `construction.md`：整体架构说明
- `101-solutions.md`：基于当前实现整理的解法说明

如果你准备改动部署、请求流、端口使用或特殊协议相关内容，建议先读 `construction.md`。

## 本地运行与测试

最基础的启动方式：

```bash
npm install
npm start
```

运行测试：

```bash
npm test
```

需要注意的是，这个项目并不是所有内容都只依赖普通本地静态访问。部分关卡和测试依赖后端行为、HTTPS、WebSocket、Trailer 或 HTTP/2 特性，所以“页面能打开”不等于“玩法完整可用”。

## 关于新关卡贡献

如果你想贡献新关卡，建议先不要直接提交完整实现，而是先说明以下内容：

- 这一关主要考什么
- 玩家需要观察什么
- 它和前面关卡有什么区分
- 是否依赖后端或特殊协议
- 是否会与已有玩法重复
- 预期难度大概在哪个位置

这个项目更需要“方向明确、表达自然、能放进整体节奏里”的关卡，而不只是单独成立的一道题。
