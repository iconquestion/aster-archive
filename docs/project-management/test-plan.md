# 测试计划文档

本文档用于记录 `Aster Archive` 第一阶段的测试目标、测试范围、测试方法、通过标准与执行安排。

本文件描述的是项目当前采用的测试思路与质量控制方式，不用于承载下一阶段的改进设想。后续测试能力扩展另见独立路线文档。

## 测试目标

第一阶段测试工作的目标如下：

- 验证第 `01` 关至第 `24` 关主线中的关键链路是否可用
- 验证需要后端参与的关卡行为是否与设计意图一致
- 验证 WebSocket、HTTPS Trailer 等特殊协议能力是否仍可工作
- 验证关键运行组件如入口关闭逻辑、日志初始化、健康检查等是否稳定
- 在不显著抬高维护成本的前提下，为后续迭代提供可重复的回归基础

## 测试范围

### 纳入当前测试计划的内容

- 站点健康检查接口
- 入口层优雅停机逻辑
- 日志目录与日志轮转初始化
- 关键静态线索关卡
- 关键 HTTP 接口关卡
- 关键认证、Cookie、CSRF 相关关卡
- WebSocket 与 HTTPS Trailer 等协议专项关卡
- 第 `25` 关与第 `26` 关已存在的后端接口与状态管理逻辑

### 当前不作为自动化主覆盖对象的内容

- 纯视觉比对型关卡
- 强依赖浏览器运行时状态的关卡
- 强依赖浏览器反调试或本地存储行为的关卡
- 需要更底层 HTTP/2 客户端细节才能完整验证的场景

这类内容并非不重要，而是暂时不放入第一阶段的自动化主覆盖范围，当前主要依赖人工复核、显式 `skip` 标记和后续测试路线规划来管理。

## 测试策略

项目当前采用“按玩法与技术风险分层”的测试策略，而不是仅按模块文件划分。

### 1. 基础运行层测试

对应内容：

- 入口关闭逻辑
- 日志初始化
- 健康检查接口
- 配置加载

测试目的：

- 确保服务具备基本可启动、可关闭、可观测能力
- 减少运行期故障定位难度

### 2. HTTP 主线集成测试

对应内容：

- 静态页面中的关键线索
- 资源文件中的隐藏路径
- 普通接口响应头、查询参数、方法分支
- Cookie、Basic Auth、表单登录、语言分支等后端逻辑

测试目的：

- 确保主线路径不会因局部改动而断裂
- 直接验证“题面提示是否仍能导向正确解法”

### 3. 协议专项测试

对应内容：

- WebSocket upgrade
- HTTPS Trailer

测试目的：

- 覆盖普通进程内 HTTP 测试无法验证的协议行为
- 降低部署层变更带来的特殊协议回归风险

### 4. 状态与存储型关卡测试

对应内容：

- 第 `25` 关会话隔离与恢复流程
- 第 `26` 关棋盘状态、会话 Cookie、库存变更与解锁流程

测试目的：

- 验证落盘状态与会话隔离是否正确
- 验证复杂关卡接口在主成功路径和主要失败路径上都能保持稳定

## 测试结构

当前自动化测试主要位于以下目录：

- `test/index.test.js`
- `test/logger.test.js`
- `test/http/status.test.js`
- `test/http/levels.test.js`
- `test/protocol/protocol.test.js`
- `test/helpers/createTestApp.js`

各部分分工如下：

### `test/index.test.js`

验证进程信号、未处理异常与优雅停机逻辑。

### `test/logger.test.js`

验证日志目录创建、轮转 transport 注册和级别分流。

### `test/http/status.test.js`

验证基础健康检查接口可用。

### `test/http/levels.test.js`

作为主线测试主体，覆盖：

- 配置与基础运行时数据
- 01-14 关中的关键静态与 HTTP 玩法
- 16-25 关中的关键 HTTP、Cookie、语言分支、Service Worker 相关行为
- 26 关 HTTP API 的会话、状态与业务规则

### `test/protocol/protocol.test.js`

覆盖：

- 15 关 WebSocket 连接
- 17 关 HTTPS Trailer

### `test/helpers/createTestApp.js`

提供静默 logger、随机端口 server、测试运行时复用和协议测试装配能力。

## 当前已覆盖的主要测试点

结合现有测试代码，当前测试计划已经覆盖以下重点：

- 01-10 关中的静态页面线索、资源线索、响应头、方法切换与查询参数分支
- 12-14 关中的登录、Cookie、Basic Auth 与隐藏页面链路
- 16、18、20、22、24 关中的关键 HTTP 行为
- 15、17 关中的协议能力
- 25 关中的会话隔离、提交与恢复逻辑
- 26 关中的棋盘初始化、会话复用、库存变化、业务失败路径与取 flag 规则
- 运行期入口与日志系统

## 当前显式留白

现有测试中，以下关卡或场景采用了显式留白策略：

- 11 关：多步解码流程
- 19 关：字体映射与视觉比对
- 21 关：HTTP/2 Early Hints 场景
- 23 关：浏览器反调试与 localStorage 场景

这些留白不会被视为“忘记测试”，而是当前测试计划中的明确边界。

## 测试方法

### 1. 进程内集成测试

适用内容：

- 普通 HTTP 路由
- 页面与静态资源
- 认证、Cookie、CSRF 与表单逻辑

方法说明：

- 使用 `supertest`
- 在内存中装配 Express app
- 通过测试辅助代码复用配置和静默 logger

### 2. 真实本地服务测试

适用内容：

- WebSocket upgrade
- HTTPS Trailer
- 需要真实监听端口的协议行为

方法说明：

- 启动随机本地端口 server
- 使用真实 `http`、`https`、`ws` 客户端访问
- 测试结束后统一关闭 server

### 3. 状态落盘测试

适用内容：

- 第 `25` 关
- 第 `26` 关

方法说明：

- 使用临时目录承载测试状态文件
- 每个测试或测试组使用隔离目录
- 测试结束后清理临时数据

### 4. 显式跳过策略

适用内容：

- 当前自动化手段成本较高或收益不稳定的关卡

方法说明：

- 通过 `test.skip` 保持留白可见
- 避免“未覆盖”被误解为“已经覆盖”
- 为后续改进预留明确入口

## 通过标准

第一阶段测试工作的通过标准如下：

- `npm test` 可作为统一主测试入口
- 基础运行层测试通过
- 已纳入自动化范围的主线关卡测试通过
- 协议专项测试通过
- 不允许因测试自身设计问题导致长期挂起或明显内存失控
- 显式留白场景必须保持清晰标识，不混入通过结果

## 执行方式

项目当前统一使用：

```bash
npm test
```

当前 Jest 运行策略包括：

- `bail: 1`
- `testTimeout: 10000`
- `maxWorkers: 25%`
- `workerIdleMemoryLimit: 512MB`

这些设置用于控制失败反馈速度、执行时间与内存占用。

## 测试数据与环境要求

### 环境要求

- 使用项目要求的 Node 版本
- 正确配置 `config/.env`
- 提供有效 TLS 证书路径，供协议测试启动 HTTPS 与 HTTP/2 服务

### 数据要求

- 第 `12` 关每日口令依赖运行日生成规则
- 第 `25` 关、第 `26` 关测试依赖临时目录与临时状态文件
- 协议测试依赖本地随机端口，不使用固定端口

## 测试计划与项目风险的关系

当前测试计划主要用于缓解以下风险：

- 特殊协议能力与部署环境不一致
- 关卡改动误伤既有解法链路
- 安全中间件或配置变更影响关卡行为
- 状态型关卡的会话隔离与落盘异常
- 测试运行资源占用过高

## 第一阶段测试结论口径

从第一阶段范围出发，当前测试计划的核心结论应表述为：

项目已经建立起覆盖关键主线路径、关键后端行为和关键协议能力的自动化测试基础；对浏览器专属、视觉型或更底层协议细节的进一步覆盖，保留在后续测试改进路线中推进。
