# 测试改进路线

本文档用于记录 `Aster Archive` 后续阶段的测试改进方向。

本文件不改变当前测试计划的边界，也不将未来设想提前写入第一阶段既有测试结论；它只负责说明下一阶段哪些测试能力值得扩展、为什么值得扩展，以及这些扩展是否具备可行性。

## 文档目标

- 记录当前自动化测试之外的主要留白
- 评估哪些改进方向具有较高收益
- 为下一阶段质量提升提供清晰路线

## 当前留白来源

现有测试已经覆盖了大量关键 HTTP 与协议行为，但仍存在以下类型的留白：

- 浏览器专属玩法
- 视觉映射或字体识别玩法
- 需要更细粒度 HTTP/2 客户端能力的场景
- 更完整的跨层回归验证

这些留白并不意味着现有测试体系无效，而是说明项目已经走到了“需要更专业测试工具才能继续扩展覆盖面”的阶段。

## 改进方向总览

| 编号 | 方向                             | 预期收益 | 实施成本 | 建议阶段           |
| ---- | -------------------------------- | -------- | -------- | ------------------ |

| T2   | 为 HTTP/2 场景补充专项客户端测试 | 中       | 中       | 下一阶段优先       |
| T3   | 建立人工回归清单并文档化         | 中       | 低       | 可立即纳入         |

| T6   | 引入端到端烟雾测试               | 中       | 中       | 下一阶段后半段     |

## 重点改进方向

### T1 引入浏览器自动化测试

#### 改进目的

用于覆盖以下类型的关卡或行为：

- 11 关多步页面解码流程
- 19 关字体映射与视觉型验证
- 23 关浏览器反调试与 localStorage 相关行为
- Service Worker 注册和页面脚本联动

#### 理论可行性评估

该方向具备较强可行性，原因如下：

- 项目中已有大量基于真实页面的静态入口，适合浏览器级访问
- 当前自动化留白主要集中在浏览器能力，而不是后端不可控逻辑
- 现代浏览器自动化框架可以较稳定地读取页面内容、执行脚本、检查 localStorage、监听网络请求和 Service Worker 注册结果

#### 收益评估

- 能补上当前最明显的自动化留白
- 能更直接验证“玩家真实看到的页面行为”
- 能降低前端脚本或页面结构调整造成的隐性回归

#### 成本与风险

- 测试执行时间会增长
- 需要额外维护浏览器环境
- 某些反调试玩法可能需要设计专门的测试入口或非默认策略

#### 结论

该方向适合作为下一阶段优先事项纳入执行。

## T2 为 HTTP/2 场景补充专项客户端测试

### 改进目的

主要用于补足第 `21` 关相关的：

- HTTP/2 连接建立
- 103 Early Hints
- 资源预加载提示
- 独立 stream 行为

### 理论可行性评估

该方向具备可行性，原因如下：

- 项目已经有独立的 HTTP/2 server 实现
- 现有后端结构已经将该能力集中在单独 server 层
- Node 本身具备可用于测试 HTTP/2 stream 的基础能力

### 收益评估

- 能把目前最典型的协议留白补上
- 能进一步验证部署层与实现层之间的一致性假设

### 成本与风险

- 调试难度高于普通 HTTP 测试
- 需要更细粒度处理 early hints 与 stream 事件
- 维护成本高于一般接口测试

### 结论

该方向适合作为下一阶段优先事项，但优先级略低于浏览器自动化测试。

## T3 建立人工回归清单并文档化

### 改进目的

对当前尚未自动化覆盖、但又不应完全依赖记忆的关卡，建立一份轻量人工回归清单。

### 理论可行性评估

该方向成本低、收益稳定，可立即实施。

### 收益评估

- 能让“当前留白”变成“有明确检查方式的留白”
- 能减少仅靠个人记忆回归带来的遗漏

### 结论

该方向适合立即纳入下一阶段工作。

## T4 收敛关卡级测试矩阵

### 改进目的

将“哪些关卡已经自动化覆盖、哪些关卡采用人工回归、哪些关卡暂时留白”整理成一份测试矩阵。

### 理论可行性评估

现有测试和题解信息已经足以支持整理该矩阵，实施成本较低。

### 收益评估

- 更容易判断测试投入是否失衡
- 更容易在新增关卡后发现测试缺口

### 结论

该方向适合与文档维护同步推进。

## T5 增加关键安全回归分组

### 改进目的

对 Cookie、CSRF、认证、会话隔离等关卡建立更清晰的安全回归集合。

### 理论可行性评估

项目现有测试已经具备相当多的安全相关断言，继续整理和补强的成本较低。

### 收益评估

- 有助于在安全依赖升级时定向回归
- 有助于降低“安全增强误伤玩法”的风险

### 结论

该方向适合在下一阶段与常规回归并行推进。

## T6 引入端到端烟雾测试

### 改进目的

从首页开始，对少量关键主线节点做端到端冒烟验证。

### 理论可行性评估

可行，但应控制范围，不宜一开始就做全量通关自动化。

### 收益评估

- 能快速发现入口失效、页面不可达、主链断裂等高影响问题
- 能作为部署后的轻量验收手段

### 成本与风险

- 若做成全量通关，维护成本会快速升高
- 某些解谜步骤不适合完全脚本化

### 结论

适合作为下一阶段后半段能力，引入少量关键路径烟雾测试即可。

## 下一阶段建议事项

综合收益、成本与项目结构，建议下一阶段优先执行以下事项：

1. 引入浏览器自动化测试能力，用于覆盖 11、19、23 等当前留白场景。
2. 为 21 关补充 HTTP/2 专项测试，验证 Early Hints 与 stream 行为。
3. 建立人工回归清单和关卡测试矩阵，明确自动化与人工分工。
4. 对 Cookie、CSRF、认证、会话隔离场景建立更清晰的安全回归分组。

## 不建议的做法

下一阶段不建议直接采取以下做法：

- 一开始就追求所有关卡全自动通关
- 在没有分类的前提下盲目扩大端到端测试规模
- 将未来测试设想直接写入当前测试计划结论
- 为了提升覆盖率数字而引入维护价值不高的脆弱测试

## 计划落点

这些改进方向更适合作为下一阶段质量提升与测试能力建设内容，而不是纳入第一阶段既有范围的完成判定。
