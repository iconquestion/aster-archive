# 项目范围说明

本文档用于记录 `Aster Archive` 第一阶段的正式项目范围。

本阶段范围不是对整个项目的永久承诺，而是对已经形成稳定主线、可对外说明、可用于计划管理与验收的部分进行收束。

## 文档目的

- 明确第一阶段项目到底要交付什么
- 将“已完成主线”和“后续扩展方向”区分开
- 为里程碑、项目进度、验收口径提供统一边界

## 第一阶段正式范围

第一阶段项目范围定义为：

`Aster Archive` 自起始页开始，到第 `24` 关为止的完整 Web 解谜主线。

这一范围包含：

- 起始说明页 `00-letsstart`
- 第 `01` 关至第 `24` 关的页面内容、静态资源与必要的后端逻辑
- 与上述关卡直接相关的接口、协议行为与测试
- 支撑这条主线运行所需的站点首页、通用服务、日志、配置和部署说明

## 范围内交付物

第一阶段范围内的交付物主要包括以下内容。

### 1. 玩家可游玩的主线内容

- 首页与起始入口页
- 第 `01` 关至第 `24` 关的页面与资源
- 支撑各关卡解谜过程所需的前后端交互
- 维持题面叙事连续性的必要文本与设定材料

### 2. 后端与运行支撑

- Node / Express 应用入口与公共服务模块
- 与关卡相关的 API 路由
- HTTPS、WebSocket、Trailer、HTTP/2 等在范围内关卡中已经实际使用到的能力
- 环境变量配置、日志记录与基础运行保护

### 3. 工程化与维护资料

- `README.md` 项目简介
- `docs/architecture/system-architecture.md` 架构说明
- `docs/design/ui-design-guidelines.md` 页面设计说明
- `docs/gameplay/level-solutions.md` 当前实现对应的题解
- `CONTRIBUTING.md`、`SECURITY.md` 等协作文档
- 自动化测试、Lint、格式化与基础 CI 配置

## 范围边界

为了避免项目计划失焦，第一阶段仅以“主线已开放至第 24 关”为完成边界。

这意味着：

- 是否存在第 `25` 关或第 `26` 关相关代码，不影响第一阶段范围定义
- 后续章节、新玩法或大规模世界观扩展，不计入第一阶段必交内容
- 对现有主线之外的试验性实现，可以继续保留在仓库中，但不作为第一阶段验收前提

## 明确不纳入第一阶段验收的内容

以下内容不纳入第一阶段正式验收范围：

- 第 `25` 关及之后的关卡
- 仅处于实验、打磨或局部重构中的新章节
- 自动化生成开发动态、自动汇总变更日志等增强型后台能力
- 更大规模的内容运营、社区化功能或账号体系
- 将项目改造为通用平台、编辑器或关卡生产工具

## 第 25 关及之后内容的定位

仓库中已包含第 `25` 关与第 `26` 关的相关代码与实现准备。

这些内容应视为：

- 第一阶段完成后的自然延伸方向
- 后续章节的探索性实现
- 为下一阶段或未来扩展预留的内容基础

它们可以继续开发，但不应反向改变第一阶段“1 至 24 关”为主范围的既定边界。

## 第一阶段完成判定

若从项目计划视角判断第一阶段是否达到预期，至少应满足以下条件：

- 首页与起始入口可正常引导玩家进入主线
- 第 `01` 关至第 `24` 关构成一条连续可游玩的主线链路
- 范围内关键后端关卡具备基本自动化测试或可复核的行为说明
- 核心运行文档、架构文档与维护文档齐备
- 对外描述与内部计划口径一致，均以“已开放至第 24 关”为阶段边界

## 后续扩展原则

第一阶段范围冻结后，后续扩展建议遵循以下原则：

- 新关卡按新阶段或新版本单独规划
- 不为了追求章节数量而频繁改写既有主线边界
- 新内容如影响架构、测试或叙事节奏，应同步更新里程碑与进度文档
