# 产品需求说明

本文档定义 `Aster Archive` 第一阶段的产品需求基线，用于统一项目目标、范围、用户体验和阶段性交付口径。

## 1. 项目概述

### 1.1 项目名称

`Aster Archive`

### 1.2 项目定位

`Aster Archive` 是一个以网页前后端交互为主要谜题载体的 Web 闯关解谜项目。

玩家通过访问页面、阅读源码、观察静态资源、分析请求响应与协议行为，在一座神秘档案馆的叙事框架中逐步寻找下一关入口。

### 1.3 项目目标

- 提供一条从起始页出发、连续可游玩的 Web 解谜主线
- 让玩家在解谜过程中接触现代 Web 基础知识与部分协议特性
- 通过档案馆叙事、页面文本与静态物料建立统一世界观氛围
- 在保持个人项目可维护性的前提下，为后续章节扩展预留空间

## 2. 背景与立项动机

当前常见 WebCTF 更偏重漏洞利用、攻防技巧或高难度技术对抗，而本项目希望建立一种更偏探索、观察与联想的 Web 解谜体验。

项目以“档案馆探索”作为叙事外壳，将页面源码、注释、资源文件、HTTP 头、Cookie、认证、WebSocket、Trailer、HTTP/2 等能力转译为玩家可感知的谜题线索，使知识点与叙事动作保持一致。

## 3. 用户与使用场景

### 3.1 目标用户

- 对网页解谜、网页交互或轻量 CTF 感兴趣的玩家
- 具备基础浏览器使用能力，希望通过实战加深 Web 理解的学习者
- 希望从页面、资源、请求与协议角度设计或研究谜题的开发者

### 3.2 核心使用场景

- 玩家从首页了解项目定位，并进入起始说明页
- 玩家按照关卡提示，在浏览器或开发者工具中寻找下一关 flag
- 玩家在卡关时查看当前关卡提示，继续推进主线
- 玩家在需要时查看项目更新动态，以了解网站进展与开放范围

## 4. 产品愿景与体验原则

### 4.1 愿景

构建一座兼具古典档案馆气质与现代 Web 交互特征的可探索站点，让玩家在“寻找下一关入口”的持续驱动下完成一次由浅入深的解谜旅程。

### 4.2 体验原则

- 线索必须真实存在于网页系统的某个可观测层面
- 关卡之间保持明确、连续的推进关系
- 谜题难度应逐步提升，但不以纯技术门槛取代探索乐趣
- 页面叙事、视觉气质与解法提示应相互呼应
- 对需要特殊协议能力的关卡，应允许玩家直接触达相应服务能力

## 5. 第一阶段范围

### 5.1 范围定义

第一阶段范围定义为：

从起始页 `00-letsstart` 开始，到第 `24` 关为止的完整主线体验。

### 5.2 范围内内容

- 首页、起始说明页与更新页
- 第 `01` 关至第 `24` 关的页面、静态资源与相关后端逻辑
- 支撑关卡运行的通用 Node/Express 服务
- 与主线相关的健康检查、日志、配置、测试和部署资料
- 题解、架构、设计、叙事与项目管理文档

### 5.3 范围边界

以下内容不纳入第一阶段必交范围：

- 第 `25` 关及之后的关卡开放
- 账号体系、排行榜、社区互动等平台化能力
- 独立关卡编辑器、通用题库后台或内容生产平台
- 大规模运营活动与商业化支持能力

## 6. 产品目标

### 6.1 内容目标

- 建立从第 `01` 关到第 `24` 关的连续主线
- 关卡覆盖静态线索、HTTP 行为与协议型玩法三类主要解谜形式
- 叙事文本、房间命名、页面物料与世界观保持统一口径

### 6.2 交互目标

- 首页明确传达项目定位、当前开放进度与入口位置
- 起始页清楚说明游戏规则、flag 形式和 API 约定
- 每一关都具备足够的题面信息，并在需要时提供提示
- 低风险页面在桌面端和移动端都应保持稳定可读

### 6.3 工程目标

- 服务端能够稳定提供静态资源、普通 API 与必要的特殊协议能力
- 项目具备统一测试入口、基础日志能力与清晰配置方式
- 文档能够支撑项目说明、实现理解与后续维护

## 7. 核心功能需求

### 7.1 站点入口与公开说明

- 提供站点首页，对外说明项目定位、玩法方向和当前开放进度
- 提供起始说明页，解释 flag 规则、API 约定、提示机制与基本游玩方式
- 提供每日更新动态页，整理主要开发推进与内容演进

### 7.2 主线关卡系统

- 关卡入口采用固定路径模式：`{两位编号}-{十位随机小写字符串}`
- 当前关卡必须能够提供下一关 flag
- 线索可存在于页面源码、注释、CSS、隐藏资源、接口返回、请求头、Cookie、认证流程或协议行为中
- 各关卡之间互不污染状态，玩家可将每关视作一个相对独立的站点场景

### 7.3 后端交互能力

- 对需要后端参与的关卡，统一支持 `/api/{关卡编号}/...` 风格路径
- 支持普通 HTTP 请求场景，包括查询参数、请求方法、认证、Cookie、表单、JSON 等交互
- 对特殊玩法关卡提供 WebSocket、Trailer、HTTP/2 等能力

### 7.4 提示与引导

- 每关默认提供提示入口
- 当某关存在例外规则或额外操作前提时，页面中应直接说明
- 提示文字应引导玩家观察系统行为，而非直接替代解谜过程

### 7.5 叙事与氛围表达

- 站点整体叙事应围绕“神秘档案馆探索”展开
- 文案、命名、目录、报刊、来函、限制访问材料等元素应服务于档案馆设定
- 叙事表达应支持古典、厚重、神秘而克制的总体气质

## 8. 非功能性需求

### 8.1 可用性

- 玩家应能在现代桌面浏览器中完成主要体验
- 首页、起始页与低风险关卡需具备基础响应式能力
- 页面应优先保持内容可读、操作清晰、结果区稳定

### 8.2 可维护性

- 项目结构应将静态页面、后端逻辑、测试与文档分层组织
- 关键实现应具备足够说明文档，便于后续补充新关卡或维护既有关卡
- 配置项集中管理，避免运行参数分散

### 8.3 可测试性

- 提供统一测试入口
- 关键 HTTP 行为、协议行为与运行基础能力应纳入自动化回归
- 暂不适合自动化的关卡应保留清晰边界，不与已覆盖内容混淆

### 8.4 可观测性

- 服务需具备健康检查接口
- 运行日志应支持分级记录、按天轮转与历史保留

## 9. 验收标准

第一阶段验收至少应满足以下条件：

- 首页与起始页可正常引导玩家进入主线
- 第 `01` 关至第 `24` 关构成连续可游玩的主线链路
- 范围内需要后端参与的关键关卡具备稳定服务能力
- 特殊协议关卡在既定部署方式下可以正常完成玩法
- 项目具备 README、架构、设计、题解、范围、测试等基础文档
- `npm test` 可作为统一主测试入口，并通过当前纳入范围的自动化测试

## 10. 阶段性交付物

- 可访问的站点首页、起始页与更新页
- 第 `01` 关至第 `24` 关的页面与相关接口
- Node/Express 服务端实现与配置机制
- 自动化测试脚本与 CI 基础能力
- 项目说明、架构、设计、叙事、题解与项目管理文档

## 11. 后续扩展原则

- 新关卡按独立阶段持续规划，不反向改写第一阶段边界
- 叙事扩展应与现有关卡风格和世界观口径保持一致
- 新增协议型玩法时，应同步评估部署兼容性、测试成本与文档要求
