# 产品需求说明

本文档定义 `Aster Archive` 第一阶段的产品需求基线，用于统一产品目标、目标用户、核心体验和产品级验收口径。

## 需求优先级说明

本文档中的用户需求按以下优先级评级：

- `P0`：第一阶段必须满足；若缺失，将直接影响主线可玩性、基础访问或阶段验收。
- `P1`：第一阶段应满足；若延期，不应破坏主线闭环，但会降低体验完整性、可维护性或稳定性。
- `P2`：可延后优化；主要用于增强表达、扩展体验或降低后续维护成本。

## 1. 项目概述

### 1.1 项目名称

`Aster Archive`

### 1.2 项目定位

`Aster Archive` 是一个以网页前后端交互为主要谜题载体的 Web 闯关解谜项目。

玩家通过利用HTML网页与前后端知识进行探索，包括但不限于访问页面、阅读源码、观察静态资源、分析请求响应与协议行为，在一座神秘档案馆的叙事框架中逐步寻找下一关入口。

### 1.3 项目目标

- 提供一条从起始页出发、连续可游玩的 Web 解谜主线
- 让玩家在解谜过程中接触现代 Web 基础知识与部分协议特性，在游戏中学习知识
- 通过档案馆叙事、页面文本与静态物料建立统一世界观氛围，吸引玩家兴趣，提升游戏体验
- 在保持个人项目可维护性的前提下，为后续章节扩展预留空间

## 2. 背景与立项动机

当前常见 WebCTF 更偏重漏洞利用、攻防技巧或高难度技术对抗，而本项目希望建立一种更偏探索、观察与联想的趣味性 Web 解谜体验。

项目以“档案馆探索”作为叙事外壳，将HTML网页与前后端等知识转译为玩家可感知的谜题线索，使知识点与叙事动作保持一致。

## 3. 用户与使用场景

### 3.1 目标用户

- 主要用户：具备基础前端 / HTTP 认知、愿意使用浏览器开发者工具继续探索的 Web 解谜玩家
- 次要用户：具备基础浏览器使用能力，希望通过实战加深 Web 理解的学习者
- 次要用户：希望从页面、资源、请求与协议角度设计或研究谜题的开发者

本阶段的题面表达、提示粒度、主线难度曲线和公开说明，默认优先服务“愿意主动观察系统行为的解谜玩家”；学习价值和研究价值属于在此基础上的次级收益。

### 3.2 核心使用场景

- 主要用户从首页了解项目定位，并进入起始说明页开始主线
- 主要用户按照关卡提示，结合浏览器开发者工具或第三方工具寻找下一关 flag
- 主要用户在卡关时查看当前关卡提示，继续推进主线
- 次要用户在需要时查看项目更新页，以了解当前开放范围与阶段进展

## 4. 产品愿景与体验原则

### 4.1 愿景

构建一座兼具古典档案馆气质与现代 Web 交互特征的可探索站点，让玩家在“寻找下一关入口”的持续驱动下完成一次由浅入深的解谜旅程。

### 4.2 体验原则

- 线索必须真实存在于网页系统的某个可观测层面
- 关卡之间保持明确、连续的推进关系
- 谜题难度应逐步提升，但不以纯技术门槛取代探索乐趣
- 页面叙事、视觉气质与解法提示应相互呼应
- 对需要特殊协议能力的关卡，应允许玩家直接触达相应服务能力

## 5. 第一阶段产品范围

### 5.1 范围定义

第一阶段范围定义为：

从起始页开始，到第 `26` 关为止的完整主线体验。

### 5.2 范围内内容

- 首页、起始说明页与更新页
- 第 `01` 关至第 `26` 关的可游玩关卡体验
- 支撑主线解谜所需的题面、线索、提示、叙事文本与静态物料
- 范围内关卡所需的页面、资源、接口与必要运行支撑
- 支撑玩家理解玩法与推进主线的公开说明

### 5.3 范围边界

第一阶段正式交付边界以本章“范围内内容”为准；未列入范围内内容的事项，不属于本阶段正式交付物。

正式项目范围与交付边界以 [项目范围说明](./project-scope.md) 为准。

## 6. 产品目标

### 6.1 内容目标

- 建立从第 `01` 关到第 `26` 关的连续主线
- 关卡覆盖静态线索、HTTP 行为与协议型玩法三类主要解谜形式
- 叙事文本、房间命名、页面物料与世界观保持统一口径

### 6.2 交互目标

- 首页明确传达项目定位、当前开放进度与入口位置
- 起始页清楚说明游戏规则、flag 形式和 API 约定
- 每一关都具备足够的题面信息，并在需要时提供提示

### 6.3 可持续体验目标

- 已开放主线应保持稳定可访问，不因后续章节开发破坏既有体验
- 玩家应能通过公开说明、页面提示和必要题解理解项目玩法
- 后续章节扩展应延续现有叙事、难度曲线和交互边界

## 7. 核心功能需求

### 7.1 站点入口与公开说明

| 编号    | 需求                                                               | 优先级 |
| ------- | ------------------------------------------------------------------ | ------ |
| `PR-01` | 提供站点首页，对外说明项目定位、玩法方向和当前开放进度。           | `P0`   |
| `PR-02` | 提供起始说明页，解释 flag 规则、API 约定、提示机制与基本游玩方式。 | `P0`   |
| `PR-03` | 提供更新页，用于发布阶段进展、开放范围和对外可见的主要内容演进。   | `P1`   |

### 7.2 主线关卡系统

| 编号    | 需求                                                                                        | 优先级 |
| ------- | ------------------------------------------------------------------------------------------- | ------ |
| `PR-04` | 关卡入口采用固定路径模式：`{两位编号}-{十位随机小写字符串}`。                               | `P0`   |
| `PR-05` | 当前关卡必须能够提供下一关 flag。                                                           | `P0`   |
| `PR-06` | 线索可存在于页面源码、注释、CSS、隐藏资源、接口返回、请求头、Cookie、认证流程或协议行为中。 | `P0`   |
| `PR-07` | 各关卡之间互不污染状态，玩家可将每关视作一个相对独立的站点场景。                            | `P1`   |

### 7.3 后端交互能力

| 编号    | 需求                                                                              | 优先级 |
| ------- | --------------------------------------------------------------------------------- | ------ |
| `PR-08` | 对需要后端参与的关卡，应提供一致、可观察的交互入口。                              | `P0`   |
| `PR-09` | 支持普通 HTTP 请求场景，包括查询参数、请求方法、认证、Cookie、表单、JSON 等交互。 | `P0`   |
| `PR-10` | 对特殊玩法关卡，应让玩家可以实际触达相应协议或交互能力。                          | `P0`   |

### 7.4 提示与引导

| 编号    | 需求                                                   | 优先级 |
| ------- | ------------------------------------------------------ | ------ |
| `PR-11` | 每关默认提供提示入口。                                 | `P1`   |
| `PR-12` | 当某关存在例外规则或额外操作前提时，页面中应直接说明。 | `P0`   |
| `PR-13` | 提示文字应引导玩家观察系统行为，而非直接替代解谜过程。 | `P1`   |

### 7.5 叙事与氛围表达

| 编号    | 需求                                                                 | 优先级 |
| ------- | -------------------------------------------------------------------- | ------ |
| `PR-14` | 站点整体叙事应围绕“神秘档案馆探索”展开。                             | `P1`   |
| `PR-15` | 文案、命名、目录、报刊、来函、限制访问材料等元素应服务于档案馆设定。 | `P1`   |
| `PR-16` | 叙事表达应支持古典、厚重、神秘而克制的总体气质。                     | `P2`   |

## 8. 非功能性需求

### 8.1 可用性

| 编号     | 需求                                                   | 优先级 |
| -------- | ------------------------------------------------------ | ------ |
| `NFR-01` | 玩家应能结合现代浏览器与必要的第三方工具完成主要体验。 | `P0`   |
| `NFR-02` | 首页、起始页与低风险关卡需具备基础响应式能力。         | `P1`   |
| `NFR-03` | 页面应优先保持内容可读、操作清晰、结果区稳定。         | `P1`   |

### 8.2 产品运行与维护约束

| 编号     | 需求                                                                                           | 优先级 |
| -------- | ---------------------------------------------------------------------------------------------- | ------ |
| `NFR-04` | 第一阶段产品设计应以单机、低成本、可长期维护的部署方式为默认前提。                             | `P0`   |
| `NFR-05` | 第一阶段完成条件不包含多节点扩容、独立缓存集群、高规格数据库或平台化改造。                     | `P0`   |
| `NFR-06` | 第一阶段内的关卡设计应与上述低成本运行前提保持一致，不依赖明显超出当前阶段维护能力的资源投入。 | `P1`   |
| `NFR-07` | 日志、缓存、构建产物与题目素材规模应控制在维护者可以长期管理的范围内。                         | `P1`   |
| `NFR-08` | 后续若出现明显超出上述运行和维护边界的新需求，应在新阶段单独评估。                             | `P1`   |

### 8.3 可测试性

| 编号     | 需求                                                     | 优先级 |
| -------- | -------------------------------------------------------- | ------ |
| `NFR-09` | 第一阶段主线应具备可复核的验收方式。                     | `P0`   |
| `NFR-10` | 关键解谜链路应能通过自动化测试或人工回归说明进行验证。   | `P1`   |
| `NFR-11` | 暂不适合自动化的关卡应保留清晰边界，不与已覆盖内容混淆。 | `P1`   |

## 10. 验收标准

第一阶段验收至少应满足以下条件：

- 首页与起始页可正常引导玩家进入主线
- 第 `01` 关至第 `26` 关构成连续可游玩的主线链路
- 范围内需要后端参与的关键关卡具备稳定服务能力
- 范围内需要状态、会话或持久化支撑的关卡具备可复核行为
- 特殊协议关卡在既定部署方式下可以正常完成玩法
- 项目具备支撑玩家理解与维护者复核的基础说明、题解和设计资料
- 第一阶段主线具备可复核的验收记录，具体测试执行口径由 [测试计划文档](./test-plan.md) 维护

## 11. 阶段性交付物

- 可访问的站点首页、起始页与更新页
- 第 `01` 关至第 `26` 关的可游玩主线内容
- 与主线体验直接相关的题面、提示、线索与叙事物料
- 范围内后端、协议或状态支撑所需的运行时状态、接口和补充测试资料
- 支撑玩家理解玩法和维护者复核主线的必要文档

## 12. 后续扩展原则

- 后续新关卡按独立阶段持续规划，不反向改写第一阶段边界
- 叙事扩展应与现有关卡风格和世界观口径保持一致
- 新增协议型玩法时，应同步评估部署兼容性、测试成本与文档要求
