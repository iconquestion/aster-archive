# 软件需求规格说明书（SRS）

本文档规定 `Aster Archive` 第一阶段的软件需求规格，用于指导实现、测试、部署与验收。

本文档采用“目标规格”立场：当当前实现与本文档不一致时，应优先将差异记录为待整改项，并以本文档作为后续实现、测试和验收基线，而不是反向降低规格。

## 1. 引言

### 1.1 目的

本文档用于明确 `Aster Archive` 第一阶段的软件行为、接口边界、数据约束、运行约束与验收口径，供开发、测试和维护共用。

### 1.2 适用范围

本规格覆盖以下内容：

- 站点首页、起始说明页与更新页
- 第 `01` 关至第 `26` 关主线所需的页面、资源与后端接口
- 健康检查、日志、配置加载与静态资源服务
- 第 `15`、`17`、`21` 关等协议型能力
- 第 `25`、`26` 关等状态型复杂交互关卡
- 与上述内容直接相关的测试和验收要求

### 1.3 文档边界

- 对纯静态关卡，本文档只保留软件层需要长期维护的高层行为约束，不逐页展开 HTML 细节。
- 对有后端或协议行为的关卡，本文档提供可实现、可测试、可验收的严格规格。
- 题面文案、叙事氛围和解法细节不在本文档中逐字冻结；其上位约束由 [产品需求说明](./product-requirements.md) 负责。

### 1.4 术语

- `flag`：下一关路径标识，格式为 `{两位编号}-{十位随机小写字符串}`。
- `主线`：从 `00-letsstart` 到第 `26` 关的连续游玩链路。
- `关卡接口`：挂载于 `/api/{关卡编号}` 下的 HTTP 或协议入口。
- `协议型关卡`：依赖 WebSocket、HTTPS Trailer、HTTP/2 等能力的关卡。
- `状态型关卡`：依赖会话、持久化或多步状态修改的关卡。
- `业务失败`：请求格式合法，但因谜题规则、库存、未通关等原因未成功完成操作。
- `请求错误`：请求缺少必要字段、字段类型错误、方法错误、路径错误或坐标不合法。

## 2. 总体描述

### 2.1 产品概述

`Aster Archive` 是一个浏览器访问型 Web 解谜应用。玩家通过阅读页面、查看源码、访问静态资源、发送 HTTP 请求、使用协议能力或操作状态型界面来获得下一关入口。

### 2.2 用户特征

- 主要用户具备基础前端 / HTTP 认知，并愿意结合浏览器开发者工具继续探索
- 次要用户可能以学习 Web 基础知识或研究谜题设计为主要动机
- 软件行为应优先保证主要用户能够理解公开说明、找到提示入口并完成主线推进

### 2.3 系统上下文

系统由以下三层构成：

- 浏览器客户端：访问页面、发起普通 HTTP 请求或协议连接
- Web 入口层：负责静态内容分发与常规 HTTP 转发
- 应用服务层：提供页面、普通 API 与范围内关卡所需的协议能力

### 2.4 第一阶段范围

第一阶段交付边界为：

- 首页、起始说明页、更新页
- 第 `01` 关至第 `26` 关完整主线
- 与主线直接相关的接口、协议能力、状态持久化、日志、配置和测试

当前主线范围之外的后续内容不属于本阶段正式验收范围，但第 `26` 关可返回进入后续内容的线索。

### 2.5 设计约束

- 第一阶段软件设计应与单机、低成本、可长期维护的部署前提保持一致
- 第一阶段完成条件不包含多节点扩容、独立缓存集群、高规格数据库或平台化改造
- 特殊协议能力不得被简化为只能通过人工伪造响应才能完成
- 配置必须通过统一配置入口加载并校验
- 状态型关卡必须保持玩家会话隔离，不得跨玩家共享可变状态

### 2.6 假设与依赖

- 部署环境可提供 HTTP、HTTPS 与 HTTP/2 监听能力
- 对第 `15`、`17`、`21` 关等协议型关卡，外部入口必须允许玩家实际访问所需能力

## 3. 需求组织与一致性规则

### 3.1 编号规则

- 功能需求使用 `FR-*`
- 非功能需求使用 `NFR-*`
- 验收要求使用 `AR-*`

### 3.2 规范优先级

- `必须`：第一阶段正式验收前必须满足
- `应`：第一阶段默认应满足；若延期，需要显式记录影响
- `可`：增强项，不作为当前阶段硬性验收前提

### 3.3 响应语义规则

除个别关卡玩法明确要求例外外，系统应遵循以下规则：

- 请求错误使用 `4xx`
- 业务失败可使用 `200` 返回结构化失败信息
- 未处理异常使用 `500`
- JSON 接口默认返回 `application/json; charset=utf-8`

### 3.4 会话与持久化规则

- 会话型接口必须使用系统签发的会话标识 Cookie 区分玩家
- 同一玩家重复访问时，应复用其已有状态，除非规则明确要求重置
- 缺失、非法或失效会话必须返回明确错误，不得静默读取其他玩家状态
- 状态文件损坏、缺失或过期时，应进入可理解的恢复路径，而不是返回模糊异常

## 4. 系统特性与功能需求

### 4.1 站点入口与静态主线

#### 4.1.1 描述

系统必须提供首页、起始页和第 `01` 至 `26` 关主线内容。纯静态关卡可以通过 HTML、CSS、注释、静态资源、目录、字体或前端脚本承载线索。

#### 4.1.2 功能需求

- `FR-01` 系统必须提供 `/` 首页，并展示项目定位、当前开放进度和主入口链接。
- `FR-02` 系统必须提供 `/00-letsstart` 起始说明页，并解释 flag 规则、基本游玩方式和必要约定。
- `FR-03` 系统必须提供 `/updates` 更新页，用于展示阶段进展、开放范围和对外可见的主要内容演进。
- `FR-04` 第 `01` 关至第 `26` 关必须构成连续主线；当前关必须能提供下一关线索。
- `FR-05` 纯静态关卡的线索可分布于页面源码、注释、CSS、字体、站点地图、隐藏文件或前端脚本，但线索必须真实存在于可观测系统层面。

### 4.2 页面提示、引导与叙事一致性

#### 4.2.1 描述

系统必须通过页面结构、提示入口、例外规则说明和统一叙事表达来支撑玩家推进主线，而不是把关键规则只隐藏在代码或接口实现中。

#### 4.2.2 功能需求

- `FR-06` 每个关卡页面应提供关卡标题和基本题面说明。
- `FR-07` 默认情况下，系统应为关卡提供提示入口；若某关不适合单独提供提示，应在公共说明或题面中明确替代方式。
- `FR-08` 当某关存在例外规则、额外操作前提或非默认访问方式时，页面或相关公开说明必须直接说明。
- `FR-09` 页面命名、主要文案和限制访问材料等表达应与“神秘档案馆探索”的叙事设定保持一致。
- `FR-10` 公开页面与关卡页面之间不得出现明显冲突的世界观口径、命名体系或开放进度表述。

### 4.3 普通 HTTP 关卡

#### 4.3.1 描述

普通 HTTP 关卡通过查询参数、请求方法、请求头、认证、Cookie、Range 或 JSON 请求体等行为向玩家暴露下一关线索。

#### 4.3.2 功能需求

- `FR-11` 后端关卡接口应优先采用 `/api/{关卡编号}/...` 路径组织。
- `FR-12` 普通 HTTP 关卡必须显式区分请求错误和业务失败。
- `FR-13` 需要玩家观察响应头、响应体、Cookie 或状态码的关卡，必须在服务端返回稳定、可重复的结果。
- `FR-14` 当关卡依赖时间、语言或请求头时，输入条件和可见反馈必须保持一致。

### 4.4 协议型关卡

#### 4.4.1 描述

第 `15`、`17`、`21` 关依赖 WebSocket、HTTPS Trailer、HTTP/2 与 `103 Early Hints` 等普通页面之外的协议能力。

#### 4.4.2 功能需求

- `FR-15` 系统必须支持第 `15` 关所需的 WebSocket 升级与消息收发。
- `FR-16` 系统必须支持第 `17` 关所需的 HTTPS Trailer 输出。
- `FR-17` 系统必须支持第 `21` 关所需的独立 HTTP/2 服务与 `103 Early Hints`。
- `FR-18` 当代理层无法稳定转发上述能力时，系统必须允许客户端直连对应应用服务端口。

### 4.5 状态型复杂关卡

#### 4.5.1 描述

第 `25`、`26` 关需要会话、文件持久化和多步状态修改，必须以比普通 HTTP 关卡更严格的规格管理。

#### 4.5.2 功能需求

- `FR-19` 第 `25` 关必须提供登录、状态查询、配置提交、快照模板读取和恢复提交能力。
- `FR-20` 第 `25` 关必须为每个会话保存独立状态，并提供过期状态的机会性清理机制。
- `FR-21` 第 `26` 关必须提供棋盘读取、取 flag、重置、单格放置、单格旋转和单格删除能力。
- `FR-22` 第 `26` 关必须为每个会话保存独立棋盘、库存和通关状态。
- `FR-23` 第 `26` 关写操作后必须重新计算通关状态，并将结果写回持久化状态。
- `FR-24` 状态型关卡必须对非法参数返回请求错误，对规则不满足的合法请求返回玩家可理解的业务失败信息。

### 4.6 运行与维护支撑

#### 4.6.1 功能需求

- `FR-25` 系统必须提供 `/api/status` 健康检查接口。
- `FR-26` 系统必须支持统一静态资源托管。
- `FR-27` 系统必须在启动时校验必要配置，缺失或非法时直接失败。
- `FR-28` 系统必须提供运行日志，并支持按日期轮转。
- `FR-29` 系统应支持优雅停机，避免测试和部署阶段出现悬挂连接。

## 5. 外部接口规格

### 5.1 公共接口约束

#### 5.1.1 通用请求约束

- 所有 JSON 写接口默认接受 `application/json`
- 表单登录接口可接受 `application/x-www-form-urlencoded`
- 未声明鉴权的接口按游客访问设计
- 除协议型接口外，未声明特殊行为的接口使用 HTTP/1.1 即可访问

#### 5.1.2 通用错误响应

当接口以 JSON 返回错误时，应优先使用以下语义：

| 场景                               | HTTP 状态码    | 最低响应要求                     |
| ---------------------------------- | -------------- | -------------------------------- |
| 缺少参数、参数类型错误、坐标不合法 | `400`          | 返回可理解错误消息               |
| 认证失败、会话失效或未授权         | `401` 或 `403` | 返回可理解错误消息               |
| 路径不存在                         | `404`          | 返回不可达或无效                 |
| 方法不允许                         | `405`          | 返回方法不允许                   |
| 未处理异常                         | `500`          | 返回通用错误消息，不泄露内部路径 |

### 5.2 健康检查接口

#### `GET /api/status`

| 项目         | 规格                               |
| ------------ | ---------------------------------- |
| 目的         | 提供基础可用性探针                 |
| 请求体       | 无                                 |
| 成功响应     | `200`，JSON：`{ "message": "ok" }` |
| 错误响应     | `500` 表示服务异常                 |
| 会话要求     | 无                                 |
| 持久化副作用 | 无                                 |

### 5.3 普通 HTTP 关卡接口

#### `GET /api/04`

| 项目     | 规格                                                                 |
| -------- | -------------------------------------------------------------------- |
| 目的     | 通过自定义响应头提供下一关线索                                       |
| 输入     | 无                                                                   |
| 成功响应 | `200`，JSON 至少包含 `message` 字段；响应头必须包含 `X-Archive-Next` |
| 错误响应 | `500`                                                                |
| 会话要求 | 无                                                                   |
| 副作用   | 无                                                                   |

#### `GET /api/05`

| 项目     | 规格                                       |
| -------- | ------------------------------------------ |
| 目的     | 通过错误方法返回拒绝信息，引导玩家切换方法 |
| 成功响应 | `400`，JSON：包含拒绝性提示                |
| 会话要求 | 无                                         |
| 副作用   | 无                                         |

#### `POST /api/05`

| 项目     | 规格                                                |
| -------- | --------------------------------------------------- |
| 目的     | 通过正确方法返回下一关线索                          |
| 请求体   | 无强制字段                                          |
| 成功响应 | `200`，JSON：包含 `message`，其中给出第 `06` 关线索 |
| 会话要求 | 无                                                  |
| 副作用   | 无                                                  |

#### `GET /api/06`

| 项目     | 规格                                       |
| -------- | ------------------------------------------ |
| 目的     | 通过查询参数分支返回普通欢迎语或下一关线索 |
| 查询参数 | `level`，当值为 `manager` 时返回线索       |
| 成功响应 | `200`，JSON：`message`                     |
| 会话要求 | 无                                         |
| 副作用   | 无                                         |

#### `GET /api/07`

| 项目     | 规格                                             |
| -------- | ------------------------------------------------ |
| 目的     | 根据 `location` 查询参数返回场景文案             |
| 查询参数 | `location`，允许缺失；特定值返回第 `08` 关线索   |
| 成功响应 | `200`，JSON：`message`                           |
| 业务规则 | 缺失位置时返回“请指定”；未知位置返回“当前不开放” |
| 会话要求 | 无                                               |
| 副作用   | 无                                               |

#### `POST /api/12/login`

| 项目          | 规格                                                                                 |
| ------------- | ------------------------------------------------------------------------------------ |
| 目的          | 校验管理员用户名和当日四位密码                                                       |
| 请求体        | `username: string`，`password: string`                                               |
| 字段规则      | `username` 必须存在；`password` 必须为四位数字，且与服务端基于当日日期生成的口令一致 |
| 成功响应      | `200`，JSON：`{ "message": "登录成功" }`                                             |
| 失败响应      | 缺少字段时 `400`；用户名错误或密码错误时 `401`                                       |
| 会话要求      | 无                                                                                   |
| Cookie 副作用 | 必须写入登录 Cookie；Cookie 应为 `HttpOnly`，并限制到关卡所需路径                    |
| 持久化副作用  | 无文件落盘；可使用短周期运行时缓存保存当日口令                                       |

#### `GET /api/12/get_room_info`

| 项目     | 规格                                                                      |
| -------- | ------------------------------------------------------------------------- |
| 目的     | 在正确 `room_id` 和有效 Cookie 下返回下一关线索                           |
| 查询参数 | `room_id: number`                                                         |
| 成功响应 | `200`，JSON：当 `room_id = 13` 且 Cookie 有效时，`message` 返回下一关线索 |
| 失败响应 | 未授权时 `401`；房间号不匹配时 `200` 返回无此房间提示                     |
| 会话要求 | 必须携带有效登录 Cookie                                                   |
| 副作用   | 无                                                                        |

#### `POST /api/14/login`

| 项目     | 规格                                               |
| -------- | -------------------------------------------------- |
| 目的     | 使用 Basic Auth 校验管理员凭据并返回下一关线索     |
| 输入     | `Authorization: Basic <base64(username:password)>` |
| 字段规则 | 必须按关卡预设凭据规则校验 Basic Auth 信息         |
| 成功响应 | `200`，JSON：`message` 返回第 `15` 关线索          |
| 失败响应 | `401`；响应头必须带出认证提示头                    |
| 会话要求 | 无                                                 |
| 副作用   | 无                                                 |

#### `GET /api/16`

| 项目     | 规格                                                                |
| -------- | ------------------------------------------------------------------- |
| 目的     | 根据未来时间点与协议提示返回线索                                    |
| 查询参数 | `timepoint: number`                                                 |
| 请求头   | 可携带 `X-Forwarded-Http3` 作为提示输入                             |
| 成功响应 | `200`，JSON：`message`；响应头必须回写 `X-Forwarded-Http3` 判定结果 |
| 失败响应 | `400`，当 `timepoint` 缺失或非法                                    |
| 业务规则 | `timepoint` 不大于当前年份时返回过去提示；未来时间点返回下一关线索  |
| 会话要求 | 无                                                                  |
| 副作用   | 无                                                                  |

#### `GET /api/18`

| 项目     | 规格                                                                                       |
| -------- | ------------------------------------------------------------------------------------------ |
| 目的     | 通过受限 Range 分段读取线索文本                                                            |
| 请求头   | `Range: bytes=start-end`                                                                   |
| 成功响应 | 无 `Range` 时可返回普通提示；合法分段请求返回 `206`，并带 `Accept-Ranges`、`Content-Range` |
| 失败响应 | `416`，用于格式错误、贪婪范围、越界范围或单次范围过大                                      |
| 业务规则 | 不允许 suffix range 和 open-ended range；单次最大范围宽度为关卡约束值                      |
| 会话要求 | 无                                                                                         |
| 副作用   | 无                                                                                         |

#### `POST /api/20`

| 项目     | 规格                                                             |
| -------- | ---------------------------------------------------------------- |
| 目的     | 对玩家猜测返回完全匹配和部分匹配数量                             |
| 请求体   | `guess: string`                                                  |
| 成功响应 | `200`，JSON：必须包含 `message`、`exact`、`partial`、`isCorrect` |
| 失败响应 | `400`，当 `guess` 缺失                                           |
| 业务规则 | 当猜测完全正确时，`isCorrect` 为 `true`，并返回下一关线索        |
| 会话要求 | 无                                                               |
| 副作用   | 无                                                               |

#### `GET /api/22`

| 项目     | 规格                                                                           |
| -------- | ------------------------------------------------------------------------------ |
| 目的     | 根据 `Accept-Language` 首选语言返回本地化文案                                  |
| 输入     | `Accept-Language` 请求头                                                       |
| 成功响应 | `200`，JSON：`message`                                                         |
| 业务规则 | 首选语言包含 `en` 时必须返回包含第 `23` 关线索的英文文案；否则返回默认中文文案 |
| 会话要求 | 无                                                                             |
| 副作用   | 无                                                                             |

### 5.4 协议型关卡接口

#### `WS /api/15/challenge`

| 项目           | 规格                                                               |
| -------------- | ------------------------------------------------------------------ | ---- | ---- | -------- |
| 目的           | 提供迷宫初始化、移动判定和通关反馈                                 |

### 5.6 第 26 关接口

#### 状态模型

第 `26` 关会话状态至少包含以下字段：

| 字段        | 类型                                          | 说明                          |
| ----------- | --------------------------------------------- | ----------------------------- |
| `width`     | `number`                                      | 棋盘宽度，第一阶段固定为 `10` |
| `height`    | `number`                                      | 棋盘高度，第一阶段固定为 `10` |
| `source`    | `{x:number,y:number}`                         | 起点                          |
| `targets`   | `Array<{x:number,y:number}>`                  | 终点列表                      |
| `cells`     | `Tile[][]`                                    | 二维棋盘                      |
| `inventory` | `{straight:number, elbow:number, tee:number}` | 当前库存                      |
| `solved`    | `boolean`                                     | 当前是否已通关                |

`Tile` 至少支持以下形态：

| `tileType` | 额外字段                          | 说明         |
| ---------- | --------------------------------- | ------------ |
| `empty`    | 无                                | 空格         |
| `source`   | 无                                | 起点         |
| `target`   | 无                                | 终点         |
| `blocker`  | 无                                | 不可操作障碍 |
| `pipe`     | `pipeType`、`rotation`、`locked?` | 水管格       |

#### Cookie 约束

| 项目     | 规格                               |
| -------- | ---------------------------------- |
| 格式     | 32 位十六进制字符串                |
| 属性     | 必须为 `HttpOnly`，`SameSite=Lax`  |
| 会话创建 | 首次读取棋盘或旧会话失效时自动创建 |

#### 业务规则

- 棋盘固定为 `10 x 10`
- `pipeType` 仅允许 `straight`、`elbow`、`tee`
- `rotation` 仅允许 `0`、`90`、`180`、`270`
- `PUT` 只能在空格创建玩家水管
- `PATCH` 只能修改已有玩家水管的方向，忽略额外字段
- `DELETE` 只能删除已有玩家水管
- `locked` 水管与 `blocker` 不可修改
- 写操作完成后必须重新计算 `solved`
- 只有所有终点与起点连通时，`solved` 才为 `true`

#### 响应体约束

除 `GET /flag` 外，本关接口应优先使用以下响应结构：

| 字段      | 类型      | 说明           |
| --------- | --------- | -------------- |
| `success` | `boolean` | 是否成功       |
| `message` | `string`  | 玩家可理解消息 |
| `solved`  | `boolean` | 当前是否通关   |

`GET /board` 额外返回：

| 字段        | 类型     | 说明           |
| ----------- | -------- | -------------- |
| `board`     | `object` | 序列化后的棋盘 |
| `inventory` | `object` | 当前库存       |

- `NFR-10` 不得在错误响应中泄露内部文件路径、证书路径或其他敏感部署细节
- `NFR-11` 状态文件路径必须使用白名单会话标识映射，避免路径穿越

### 7.4 可维护性

- `NFR-12` 代码、静态资源、测试与文档必须分层组织
- `NFR-13` 后端关卡逻辑应按关卡拆分，便于增量维护
- `NFR-14` 文档必须能支持实现、测试、部署和问题定位
- `NFR-15` 状态型关卡的业务规则必须保持可追踪，不得散落在大量隐式工具函数中

### 7.5 可测试性

- `NFR-16` 系统必须提供 `npm test` 作为统一自动化测试入口
- `NFR-17` 健康检查、关键 HTTP 行为、日志初始化和关键协议能力必须纳入自动化覆盖
- `NFR-18` 第 `25`、`26` 关必须有会话隔离、持久化和主要失败路径回归测试

### 8.3 人工验证最小步骤

#### `AR-09` 第 21 关 HTTP/2 与 Early Hints

执行人应使用支持 HTTP/2 与 `103 Early Hints` 的客户端或抓包工具，至少完成以下步骤：

1. 请求第 `21` 关对应的 HTTP/2 入口。
2. 确认服务端在正式 `200` 响应前先返回 `103 Early Hints`。
3. 确认 `103` 中包含 `Link: <analytics.js>; rel=preload; as=script`。
4. 继续请求与第 `21` 关同属一组 HTTP/2 入口的 `analytics.js`。
5. 确认脚本内容中包含第 `22` 关线索。

#### `AR-11` 协议入口未被部署层阻断

执行人应至少完成以下连通性检查：

1. 使用 WebSocket 客户端连接 `/api/15/challenge`，确认连接建立后收到欢迎消息，并能继续发送 `{"action":"init"}` 得到地图尺寸反馈。
2. 使用支持 Trailer 的 HTTPS 客户端请求第 `17` 关对应的 HTTPS 入口，确认普通响应头包含 `Trailer` 声明，响应结束时可读取到包含第 `18` 关线索的 Trailer。
3. 使用支持 HTTP/2 与 Early Hints 的客户端请求第 `21` 关对应的 HTTP/2 入口，确认可看到 `103 Early Hints`，且后续可访问 `analytics.js`。

#### `AR-12` 提示与叙事一致性

执行人应至少完成以下页面级检查：

1. 抽查首页、起始页和若干代表性关卡页面，确认页面存在关卡标题或等价题面标识。
2. 抽查存在例外规则或特殊访问方式的关卡，确认题面、提示或公开说明已直接说明相关前提。
3. 抽查提示文本，确认提示是在引导观察系统行为，而不是直接给出完整答案。
4. 抽查首页、起始页、关卡页和限制访问材料，确认命名、世界观表述和开放进度不存在明显冲突。

### 8.4 人工验证责任和范围

- 责任主体：第一阶段维护者或指定验收执行人
- 验证范围：真实协议连通性、浏览器环境特性、自动化未覆盖的可玩性检查
- 记录要求：至少记录执行日期、环境、步骤、结果和异常现象
- 失败处理：人工验证失败不得以“自动化未覆盖”为由忽略，必须回到实现或部署层整改
