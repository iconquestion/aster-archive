# 软件需求规格说明书（SRS）

本文档规定 `Aster Archive` 第一阶段的软件需求规格，用于指导实现、测试、部署与验收。

## 1. 引言

### 1.1 目的

本文档用于明确 `Aster Archive` 第一阶段的软件行为、接口边界、运行约束与质量要求，作为开发、测试和维护的统一依据。

### 1.2 适用范围

本规格覆盖以下内容：

- 站点首页、起始说明页与更新页
- 第 `01` 关至第 `24` 关主线所需的页面、资源与后端接口
- 支撑主线运行的 Node 应用、静态资源服务、日志与配置能力
- 与上述内容直接相关的测试与验收口径

第 `25` 关及之后的实现不属于本规格的必交范围，但可作为后续扩展的参考实现。

### 1.3 术语

- `flag`：下一关路径标识，格式为 `{两位编号}-{十位随机小写字符串}`
- `关卡`：一次独立的网页解谜场景
- `主线`：从 `00-letsstart` 到第 `24` 关的连续游玩链路
- `Node 应用层`：负责静态资源、API 与部分特殊协议能力的服务实现

## 2. 总体描述

### 2.1 产品视角

`Aster Archive` 是一个浏览器访问型 Web 解谜应用。玩家以普通访客身份进入站点，通过浏览页面、分析静态文件与请求响应、调用接口或使用特殊协议能力来寻找下一关路径。

### 2.2 用户特征

- 能使用现代浏览器访问网页
- 愿意使用浏览器开发者工具观察源码、资源与网络请求
- 对 HTTP、前端资源和基础 Web 交互有一定兴趣或学习需求

### 2.3 运行环境

- 服务端：Node.js `24.14.0`
- Web 框架：Express
- 前端资源：HTML、CSS、JavaScript、Bootstrap
- 可选入口层：Nginx 反向代理
- 浏览器：支持现代 HTML、Fetch、Cookie、WebSocket、Service Worker 等能力

### 2.4 设计约束

- 静态资源以 `public/` 为主要内容层
- 后端逻辑以 `src/` 为主要实现层
- 需要特殊协议的关卡不得强依赖通用代理链路完成全部行为
- 配置必须通过统一环境变量入口加载并校验
- 第一阶段默认部署目标为当前云主机单机环境，已确认硬件规格为 `2 vCPU`、`1.6 GiB RAM`、`4 GiB Swap`、`40 GiB` 系统盘
- 产品设计与实现必须以单进程 Node/Express 服务可稳定运行作为前提，不依赖多节点集群、独立缓存集群或高规格数据库支撑
- 在第一阶段范围内，服务应能够承载约 `30` 名同时在线玩家的正常游玩，包括静态资源访问、常规 API 请求以及少量协议型关卡连接
- 单关卡实现应优先采用轻计算、低内存占用方案，避免长时间阻塞事件循环、频繁创建大对象、超大响应体或高开销实时广播
- 特殊协议或高交互关卡必须限制连接数量、消息频率与单次资源消耗，避免单个关卡挤占整机 CPU 与内存
- 日志、缓存、构建产物与题目素材总量应控制在当前磁盘容量可长期维护的范围内，避免依赖额外挂载存储才能完成第一阶段部署
- 如后续玩法设计预计超过上述单机资源边界，应视为范围外增强需求，并同步补充扩容方案、容量评估与运维成本说明

### 2.5 假设与依赖

- TLS 证书路径可用于 HTTPS 与相关协议测试
- 部署环境允许 Node 监听 HTTP、HTTPS 与 HTTP/2 端口
- 对需要特殊端口直连的关卡，外部入口应提供可达路径

## 3. 系统特性

### 3.1 首页与起始页

#### 3.1.1 描述

系统应提供首页与起始说明页，作为玩家进入项目的统一入口。

#### 3.1.2 功能要求

- `FR-01` 系统应提供 `/` 首页，并展示项目定位、基础说明、当前开放进度和主入口链接。
- `FR-02` 系统应提供 `/00-letsstart` 起始说明页，并说明 flag 规则、API 路径约定、提示机制和基础游玩方式。
- `FR-03` 首页应提供进入主线和查看更新动态的明确入口。

### 3.2 主线关卡链路

#### 3.2.1 描述

系统应提供从第 `01` 关至第 `24` 关的连续主线，玩家在当前关中获取下一关 flag 后继续推进。

#### 3.2.2 功能要求

- `FR-04` 每个主线关卡应对应唯一页面入口路径。
- `FR-05` 每个主线关卡应包含可供玩家发现的下一关线索。
- `FR-06` 当玩家使用正确 flag 访问下一关路径时，系统应返回对应关卡页面；错误路径应表现为不可达或无效。
- `FR-07` 关卡应允许将线索分布在源码、注释、CSS、隐藏文件、静态目录、接口响应、响应头、Cookie、认证流程或协议交互中。

### 3.3 普通 HTTP 交互关卡

#### 3.3.1 描述

对于需要服务端参与的主线关卡，系统应通过标准 HTTP 能力提供可观察、可验证的行为分支。

#### 3.3.2 功能要求

- `FR-08` 后端关卡接口应优先使用 `/api/{关卡编号}/...` 路径组织。
- `FR-09` 系统应支持以查询参数、HTTP 方法、表单、JSON、认证头或 Cookie 作为关卡交互条件。
- `FR-10` 系统应返回与关卡设计一致的文本、状态或响应头结果。
- `FR-11` 系统应为需要每日口令的场景提供基于日期生成的服务端校验逻辑。

### 3.4 特殊协议关卡

#### 3.4.1 描述

系统应为少数依赖特殊协议能力的关卡提供独立支撑。

#### 3.4.2 功能要求

- `FR-12` 系统应支持 WebSocket 升级连接，以满足第 `15` 关玩法。
- `FR-13` 系统应支持 HTTPS Trailer，以满足第 `17` 关玩法。
- `FR-14` 系统应提供独立 HTTP/2 服务能力，以满足第 `21` 关玩法。
- `FR-15` 当代理层无法稳定转发对应能力时，系统应允许客户端直接访问 Node 提供的相应端口。

### 3.5 页面提示与内容呈现

#### 3.5.1 描述

系统应通过页面文本、结果区与提示区向玩家传达必要信息，同时保持统一的题面表达方式。

#### 3.5.2 功能要求

- `FR-16` 每个关卡页面应包含关卡标题和基本题面说明。
- `FR-17` 默认情况下，关卡页面应提供提示入口；若存在多级提示，可继续扩展。
- `FR-18` 页面交互结果应在稳定的结果区中展示，减少布局跳动和误读。
- `FR-19` 低风险关卡页面应遵循统一展示结构、阅读宽度和基础样式规范。

### 3.6 更新动态页

#### 3.6.1 描述

系统应提供面向访客的更新动态页，用于展示项目推进历史。

#### 3.6.2 功能要求

- `FR-20` 系统应提供 `/updates` 页面，按自然日汇总主要开发推进。
- `FR-21` 更新动态页应支持返回首页和访问仓库链接。

### 3.7 运行与维护支撑

#### 3.7.1 描述

系统应具备支撑运行、定位问题与持续维护的基础能力。

#### 3.7.2 功能要求

- `FR-22` 系统应提供 `/api/status` 健康检查接口，并返回成功状态。
- `FR-23` 系统应支持统一的静态资源托管。
- `FR-24` 系统应记录运行日志，并支持按日期轮转。
- `FR-25` 系统应在启动时校验必要配置，缺失或错误时直接失败。

## 4. 外部接口需求

### 4.1 用户接口

- 首页提供站点介绍、入口链接和进度展示
- 起始页提供规则说明和开始游戏入口
- 关卡页提供题面、提示区、按钮或输入区、结果展示区
- 更新页提供按时间排列的开发摘要

### 4.2 软件接口

- 静态资源接口：`/`、`/00-letsstart`、`/{flag}`、`/css/*`、`/js/*` 等
- 健康检查接口：`/api/status`
- 关卡接口：`/api/{关卡编号}/...`
- WebSocket 接口：`/api/15/challenge`
- HTTPS Trailer 接口：`/api/17`
- HTTP/2 相关接口：`/api/21`

### 4.3 通信接口

- HTTP：基础页面与普通 API
- HTTPS：常规站点访问与部分特殊关卡能力
- WebSocket：第 `15` 关实时交互
- HTTP/2：第 `21` 关协议玩法

## 5. 数据需求

### 5.1 静态内容数据

- 页面 HTML
- CSS、JS、字体、图片及其他线索资源
- 受限目录、隐藏文件、站点地图、订阅源等辅助线索载体

### 5.2 运行配置数据

系统至少应支持以下配置项：

- `HTTP_PORT`
- `HTTPS_PORT`
- `HTTP2_PORT`
- `APP_ORIGIN`
- `TLS_KEY_PATH`
- `TLS_CERT_PATH`

### 5.3 运行时状态数据

- 日志文件
- 每日口令缓存
- 范围内关卡运行所需的临时会话或状态数据

第 `25` 关及之后的状态数据属于后续扩展实现，不作为第一阶段 SRS 的必需数据项。

## 6. 非功能需求

### 6.1 性能

- 页面与普通 API 应在合理时间内返回结果，不得因无关额外计算造成明显阻塞
- 测试执行应控制失败反馈速度与内存占用，避免长时间挂起

### 6.2 兼容性

- 低风险页面应兼容桌面端与移动端常见视口
- 核心体验应基于现代浏览器能力设计，不要求兼容过旧浏览器

### 6.3 可靠性

- 系统应支持优雅停机
- 关键配置错误时必须显式失败，避免带病启动
- 特殊协议玩法应通过独立测试或明确验证方式维持稳定性

### 6.4 安全性

- 系统不以真实业务安全为主要目标，但应避免无意义的高风险默认配置
- 对需要 Cookie、登录或 CSRF 语义的关卡，应在实现上保持基本边界清晰

### 6.5 可维护性

- 代码、静态内容、测试与文档应分层组织
- 关卡后端逻辑应按关卡拆分，便于增量维护
- 文档应能支持项目说明、架构理解、测试执行和后续扩展

### 6.6 可测试性

- 系统应提供 `npm test` 作为统一测试入口
- 关键 HTTP 行为、日志初始化、健康检查与协议能力应具备自动化覆盖
- 暂未自动化的场景应以显式留白方式保留

## 7. 验收要求

- `AR-01` 首页、起始页和更新页可正常访问
- `AR-02` 第 `01` 关至第 `24` 关形成连续主线
- `AR-03` 健康检查接口返回正常结果
- `AR-04` 范围内关键 HTTP 关卡行为与设计一致
- `AR-05` WebSocket、Trailer、HTTP/2 等特殊玩法具备既定支撑能力
- `AR-06` 日志、配置、测试与基础文档齐备
- `AR-07` `npm test` 通过当前纳入范围的自动化测试

## 8. 追踪关系

| 产品需求           | 对应软件规格      |
| ------------------ | ----------------- |
| 站点入口与公开说明 | `FR-01` ~ `FR-03` |
| 主线关卡系统       | `FR-04` ~ `FR-15` |
| 提示与引导         | `FR-16` ~ `FR-19` |
| 更新动态           | `FR-20` ~ `FR-21` |
| 工程支撑能力       | `FR-22` ~ `FR-25` |
