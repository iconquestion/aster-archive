# Aster Archive 第一阶段系统架构说明

本文档作为 `Aster Archive` 第一阶段的正式软件架构设计文档（SAD）。

本文档采用以下立场：

- 以 [产品需求说明](../project-management/product-requirements.md) 作为产品目标与范围基线
- 以 [软件需求规格说明书](../project-management/software-requirements-specification.md) 作为软件行为、接口和验收基线
- 以当前仓库代码作为实现映射依据，用于说明现有架构如何支撑上述需求

本文档的目标不是逐页复述题面，也不是替代题解或测试计划，而是从维护者视角说明：

- 第一阶段系统为什么采用当前架构
- 系统由哪些层、模块和数据流组成
- 哪些关卡能力属于架构级能力，而不是局部页面细节
- 当前实现如何支撑部署、测试、维护与后续定位问题

## 1. 文档范围与边界

本文档覆盖 `Aster Archive` 第一阶段正式范围内、需要长期维护的架构内容，包括：

- 首页、起始说明页、更新页
- 第 `01` 关至第 `26` 关主线所需的页面、静态资源、接口和协议能力
- 健康检查、统一配置加载、日志、优雅停机等运行支撑能力
- 第 `15`、`17`、`21` 关等协议型关卡
- 第 `25`、`26` 关等状态型复杂关卡
- 与上述内容直接相关的测试和部署边界

本文档不展开以下内容：

- 纯静态关卡的逐页 HTML 结构与解法细节
- 题解、完整通关路径和谜底推导过程
- 逐条接口的低层字段穷举；这些以 SRS 为准
- 后续章节或第 `26` 关之后的扩展内容

## 2. 架构驱动因素

第一阶段系统架构主要受以下因素驱动。

### 2.1 产品驱动

- 项目需要提供一条从起始页开始、到第 `26` 关为止连续可游玩的 Web 解谜主线
- 线索必须真实存在于玩家可观察的网页、资源、请求响应或协议行为中
- 玩家应能通过页面提示、公开说明和系统行为逐步推进，而不是依赖不可见的“后台判题黑盒”
- 站点整体叙事需围绕“神秘档案馆探索”展开，页面、命名和限制访问材料之间保持统一语气

这些要求决定了系统不能只是普通静态站点，也不能只靠一组孤立 API；它必须同时支持页面表达、资源承载、后端交互和少量协议级玩法。

### 2.2 技术驱动

- 主线关卡同时覆盖纯静态线索、普通 HTTP 行为、协议型玩法和状态型复杂交互
- 部分关卡要求玩家真实接触 WebSocket、HTTPS Trailer、HTTP/2 与 `103 Early Hints`
- 部分关卡要求玩家拥有独立会话，并通过多步操作修改自己的状态

这些要求决定了系统需要：

- 统一的静态资源托管入口
- 按关卡拆分的 HTTP 路由组织
- 能直接控制 Upgrade、Trailer 和 HTTP/2 stream 的协议层实现
- 可读、可调试、可隔离的会话状态持久化方案

### 2.3 运行与维护驱动

第一阶段默认以单机、低成本、可长期维护为前提，不以多节点扩容、独立缓存集群或高规格数据库为目标。

这直接推动了以下架构选择：

- 使用 Node.js 单进程装配应用与协议服务
- 使用文件系统而不是数据库承载少量状态型关卡的持久化
- 将特殊协议能力拆到明确的独立入口，而不是引入更重的平台化组件
- 保持代码、测试和文档分层，避免后续维护时知识散落

### 2.4 质量属性驱动

第一阶段重点关注以下质量属性：

- 可维护性：模块职责清晰，代码和文档能长期被人工接管
- 可调试性：配置、日志、状态文件和协议入口具备明确边界
- 可测试性：关键 HTTP 行为和部分协议行为可自动化回归；难以自动化的能力保留人工验收边界
- 状态隔离：状态型关卡必须按玩家会话隔离，不能串读、串写其他玩家状态
- 部署兼容性：对代理层兼容性差的能力，允许玩家直连对应 Node 服务入口

## 3. 第一阶段范围与架构边界

第一阶段的正式交付边界为：

- 首页 `/`
- 起始说明页 `/00-letsstart`
- 更新页 `/updates`
- 第 `01` 关至第 `26` 关完整主线
- 与主线直接相关的后端接口、协议能力、状态持久化、日志、配置和测试支撑

从架构视角看，第一阶段可以拆为四类能力：

- 静态内容能力：用于承载首页、说明页、更新页与纯静态线索关卡
- 普通 HTTP 能力：用于承载查询参数、请求方法、头部、认证、Cookie、JSON 等行为型谜题
- 协议型能力：用于承载第 `15`、`17`、`21` 关等需要真实协议特性的玩法
- 状态型能力：用于承载第 `25`、`26` 关等依赖会话、持久化和多步修改的复杂交互

后续章节不属于本阶段正式架构范围；即使第 `26` 关返回了后续入口线索，也不代表后续内容纳入本文件维护边界。

## 4. 系统上下文与总体结构

### 4.1 系统上下文

运行时系统由以下参与方构成：

- 浏览器客户端或第三方工具：用于访问页面、抓取静态资源、发送普通 HTTP 请求，或连接协议型入口
- Web 入口层：常规情况下负责静态内容分发与普通 HTTP 转发，可由 Nginx 之类的入口承担
- Node 应用层：负责静态资源托管、普通 API、部分 HTTPS 能力，以及独立 HTTP/2 服务
- 本地文件系统：承载日志、配置文件和状态型关卡的 JSON 状态文件

### 4.2 总体结构

当前实现采用“单 Node 应用 + 多协议入口”的总体结构：

- 一个 Express 应用负责通用中间件、静态资源和大多数 HTTP 路由
- 一个 HTTP Server 承载普通站点访问，并接管 WebSocket Upgrade
- 一个 HTTPS Server 承载需要 HTTPS 访问的普通 Express 路由
- 一个独立 HTTP/2 Server 承载第 `21` 关与其专用脚本资源

这种结构的核心目标是：在保持单机和低维护成本前提下，让普通内容、协议玩法和状态型玩法都能在同一项目中稳定共存。

## 5. 逻辑架构分层

### 5.1 静态内容层

静态内容层负责：

- 首页、起始说明页与更新页
- 纯静态关卡页面
- CSS、JS、字体、图片、隐藏资源、目录索引等线索载体

这一层以 `public/` 为主要承载位置，由 Express 统一静态托管。对于第 `08` 关这类明确依赖目录索引展示的玩法，系统会在静态托管之外额外挂载目录索引能力，而不是把这类行为硬编码进页面脚本。

### 5.2 普通 HTTP 关卡层

普通 HTTP 关卡层负责承载：

- 查询参数分支
- 请求方法分支
- 请求头与响应头观察
- Cookie、表单登录、Basic Auth
- JSON 请求体与普通业务反馈

这一层的核心特征是：玩家主要通过 HTTP/1.1 风格请求观察系统行为，关卡逻辑以 `/api/{关卡编号}` 形式组织，并通过 Express Router 按关卡拆分。

### 5.3 协议型关卡层

协议型关卡层负责承载普通静态页面和常规路由之外的协议特性：

- 第 `15` 关：WebSocket Upgrade 与消息收发
- 第 `17` 关：HTTPS Trailer
- 第 `21` 关：独立 HTTP/2 服务与 `103 Early Hints`

这类关卡之所以提升到架构层处理，是因为它们往往受代理层兼容性、Node 底层协议接口和测试方式影响，不能简单视为“又一个普通 API”。

### 5.4 状态型复杂关卡层

状态型复杂关卡层负责承载需要玩家独立状态、持久化和多步修改的交互：

- 第 `25` 关：运维终端式字段查看、提交和恢复
- 第 `26` 关：水管棋盘生成、库存管理、通关判定和取 flag

这类关卡不只是“接口比较多”，而是有明确状态模型、Cookie 会话、落盘格式、恢复路径和业务规则，因此在架构上单独成层。

### 5.5 运行支撑层

运行支撑层负责非关卡本身、但对第一阶段持续运行必不可少的能力：

- `config/.env` 配置加载与校验
- `/api/status` 健康检查
- 统一日志输出与轮转
- 优雅停机
- 测试装配与本地协议测试支撑

## 6. 模块视图

### 6.1 进程入口模块

[src/index.js](/var/www/www.iconquestion.com/src/index.js) 是 Node 进程入口模块，负责：

- 加载配置
- 创建日志器
- 创建 Express 应用
- 创建 HTTP / HTTPS / HTTP/2 服务
- 注册 `SIGINT`、`SIGTERM`、`uncaughtException`、`unhandledRejection` 的优雅停机逻辑

该模块不定义具体业务路由，而是负责把运行期组件串联起来。

### 6.2 配置模块

[src/config.js](/var/www/www.iconquestion.com/src/config.js) 负责：

- 从 `config/.env` 读取运行配置
- 校验端口、主入口来源、TLS 证书路径等必要参数
- 为启动过程和测试辅助逻辑提供同一套配置解析入口

这一设计避免了“入口一套配置逻辑、测试另一套配置逻辑”的漂移风险。

### 6.3 Express 应用装配模块

[src/app.js](/var/www/www.iconquestion.com/src/app.js) 负责：

- 装配 `urlencoded`、JSON、Cookie 解析等通用中间件
- 统一处理基础 CORS 与预检请求
- 挂载 `public/` 下的静态资源
- 挂载目录索引、Bootstrap 静态资源、健康检查接口和各关卡 Router
- 定义全局错误处理

该模块只负责 Express 应用内部装配，不直接监听端口。

### 6.4 协议服务创建模块

[src/createServers.js](/var/www/www.iconquestion.com/src/createServers.js) 负责：

- 创建 HTTP、HTTPS 和独立 HTTP/2 服务
- 在底层 server 层接管第 `15` 关 WebSocket Upgrade
- 在 HTTP/2 stream 层处理第 `21` 关及其专用脚本
- 关闭服务时释放 HTTP/2 session 和空闲连接

这个模块把“协议层能力”从 Express 路由层显式分离出来，避免维护者在普通中间件栈里排查底层协议问题。

### 6.5 关卡路由模块

`src/levels/` 目录下的模块按关卡拆分后端逻辑。当前有后端逻辑的第一阶段关卡包括：

- `04`、`05`、`06`、`07`
- `12`、`14`
- `15`、`16`、`17`、`18`
- `20`、`22`
- `25`、`26`

这种拆分方式的目的不是追求抽象统一，而是让维护者在修改某一关时能直接定位到对应实现，降低误伤其他关卡解法链路的风险。

### 6.6 日志模块

[src/logger.js](/var/www/www.iconquestion.com/src/logger.js) 负责创建 Winston 日志器，并将控制台输出与文件轮转输出统一到同一套初始化流程中。

## 7. 关键运行流程

### 7.1 常规页面访问流程

常规页面访问流程如下：

1. 玩家通过浏览器访问首页、起始页、更新页或某个静态关卡页面。
2. 请求先到 Web 入口层，再转发到 Node 的普通 HTTP 服务，或直接访问 Node。
3. Express 静态资源中间件从 `public/` 返回页面或资源。
4. 玩家继续通过源码、样式、脚本、资源地址或目录结构观察线索。

这个流程覆盖了第一阶段大量纯静态或“静态为主”的关卡。

### 7.2 普通 HTTP 关卡流程

普通 HTTP 关卡流程如下：

1. 玩家或工具向 `/api/{关卡编号}` 发送请求。
2. Express 先完成通用中间件处理，如 JSON 解析、表单解析、Cookie 解析与基础 CORS。
3. 请求进入对应关卡 Router。
4. Router 根据请求方法、参数、请求头、Cookie 或认证信息执行关卡逻辑。
5. 系统返回 JSON、响应头、状态码或 Cookie，作为玩家可观察的线索反馈。

这一流程覆盖了多数后端参与但不需要特殊协议的关卡。

### 7.3 协议型关卡流程

#### 第 15 关

1. 客户端向 HTTP 服务发起 WebSocket Upgrade 请求。
2. Upgrade 不经过普通 Express 路由栈，而由底层 HTTP Server 的 `upgrade` 事件接管。
3. 第 `15` 关模块完成握手、初始化消息发送、移动消息处理和通关关闭连接。

#### 第 17 关

1. 客户端通过 HTTPS 访问第 `17` 关接口。
2. Express 路由返回正文与普通响应头。
3. 响应结束时附加 Trailer，玩家从 Trailer 中得到下一关线索。

#### 第 21 关

1. 客户端直接访问独立 HTTP/2 服务上的 `/api/21`。
2. 服务端在正式响应前先发送 `103 Early Hints`，预加载 `analytics.js`。
3. 随后返回正式 JSON 响应。
4. 客户端继续请求同一 HTTP/2 入口下的 `analytics.js`，取得后续线索。

### 7.4 状态型关卡流程

#### 第 25 关

1. 玩家登录后，系统创建会话 Cookie 和一份新的 JSON 状态文件。
2. 玩家通过状态查询、提交和恢复接口读取或修改自己的字段值。
3. 每次修改都会写回对应状态文件。
4. 系统通过会话标识保证不同玩家之间状态隔离。
5. 每次请求入口会机会性清理过期状态文件，但清理失败不会中断当前请求。

#### 第 26 关

1. 玩家首次请求棋盘时，系统检查 Cookie；无有效会话则自动创建新会话和新棋盘。
2. 棋盘状态、库存和通关状态落盘到该玩家会话文件。
3. 玩家通过 `PUT`、`PATCH`、`DELETE` 修改棋盘。
4. 每次成功写操作后，系统都会重新计算 `solved` 并落盘。
5. 只有当前会话已通关时，`GET /flag` 才会返回下一关线索。

## 8. 数据与持久化视图

### 8.1 配置数据

系统启动依赖 `config/.env` 中的配置数据，主要包括：

- `HTTP_PORT`
- `HTTPS_PORT`
- `HTTP2_PORT`
- `APP_ORIGIN`
- `TLS_KEY_PATH`
- `TLS_CERT_PATH`
- 日志轮转相关配置

这些配置在启动阶段即被严格校验；缺失、格式错误或路径不存在都会直接导致启动失败，而不是带着不完整配置继续运行。

### 8.2 日志数据

日志默认写入 `logs/` 目录，采用按日期轮转的方式持久化。

日志的主要作用是：

- 记录启动与监听信息
- 记录未捕获错误
- 辅助维护者在协议层、关闭流程和运行异常中定位问题

### 8.3 运行时短周期数据

第一阶段仅有少量运行时短周期数据。例如第 `12` 关的当日口令会基于日期生成，并允许通过进程内短期缓存避免重复计算。

这类数据不进入长期持久化系统，也不需要独立数据库。

### 8.4 第 25 关状态文件

第 `25` 关使用 `/tmp` 下的独立目录保存每个会话的 JSON 状态文件。每份状态至少包含：

- 当前字段名
- 当前字段值
- 更新时间
- 可选的最近恢复记录

状态文件的设计目标是：

- 可人工查看
- 易于调试
- 与单机部署前提相匹配
- 足以支持本关所需的多步修改和恢复流程

### 8.5 第 26 关状态文件

第 `26` 关同样使用 `/tmp` 下的独立目录保存每个会话的 JSON 棋盘状态文件。每份状态至少包含：

- 棋盘尺寸
- 起点与终点
- `cells` 二维数组
- 当前库存
- 当前是否通关

系统不会把这套状态拆成多张表或多类缓存，而是直接把完整状态按可读 JSON 落盘。原因是本关状态规模有限，且维护者更需要“可读、可调试、可人工检查”。

## 9. 会话、隔离与安全边界

### 9.1 会话标识策略

状态型关卡使用服务端签发的 Cookie 区分玩家会话。

第 `25` 关和第 `26` 关各自使用独立的 Cookie 名称，这样可以避免不同谜题之间共享状态语义，符合“各关卡尽量相互独立”的产品要求。

### 9.2 会话到文件路径的映射

系统不会直接把任意 Cookie 内容拼接成文件路径。

在第 `25` 关和第 `26` 关中，会话标识都必须先通过“32 位十六进制字符串”白名单校验，才能映射到对应 JSON 文件路径。这样做的目的是降低路径穿越、非法会话串扰和脏数据误读的风险。

### 9.3 状态失效与恢复路径

当出现以下情况时，系统会把会话视为失效或无效：

- Cookie 缺失
- Cookie 格式非法
- 状态文件不存在
- 状态文件无法读取或 JSON 无法解析

第 `25` 关在这类情况下返回“会话失效，需要重新登录”的明确错误；第 `26` 关在读取棋盘时则会自动创建新会话和新棋盘，从而为玩家提供可理解的恢复路径。

### 9.4 错误边界

全局错误处理会将未捕获异常记录到日志，并返回通用错误消息，而不是把内部路径、证书路径或堆栈细节暴露给玩家。

## 10. 部署视图与环境约束

### 10.1 部署前提

第一阶段默认采用单机部署，不要求多节点、高可用编排或独立数据库。部署环境需要满足：

- 能运行 Node.js
- 能提供 HTTP、HTTPS 与 HTTP/2 监听能力
- 能提供 TLS 证书和私钥文件
- 能访问本地文件系统中的配置、日志和临时状态目录

### 10.2 服务划分

当前实现会同时启动三类服务：

- 普通 HTTP 服务：承载大多数页面与接口，并接管第 `15` 关 Upgrade
- HTTPS 服务：承载需要 HTTPS 的普通 Express 路由，包括第 `17` 关
- 独立 HTTP/2 服务：承载第 `21` 关及其专用脚本资源

### 10.3 Web 入口层约束

Nginx 等入口层在本项目中是“可选反向代理”，而不是唯一架构核心。对于协议型关卡，需要明确以下事实：

- 第 `15` 关只有在入口层正确转发 Upgrade 头时才能通过代理工作
- 第 `17` 关的 Trailer 能力在部分代理路径下可能表现不稳定，因此允许客户端直接访问 Node HTTPS 入口
- 第 `21` 关依赖独立 HTTP/2 与 `103 Early Hints`，因此设计上不依赖普通 HTTP 反向代理出口

这也是为什么部署文档和测试文档都会单独强调协议入口边界。

## 11. 可测试性与验收支撑

### 11.1 测试分层策略

第一阶段测试采用分层策略，而不是按文件机械划分：

- 基础运行层测试：配置、日志、健康检查、优雅停机
- HTTP 主线集成测试：静态线索、普通 HTTP 接口、认证、Cookie、状态型关卡回归
- 协议专项测试：WebSocket、HTTPS Trailer
- 人工验收：当前不适合完整自动化的 HTTP/2 与浏览器特定行为

### 11.2 架构对测试的支持

当前架构通过以下方式支撑测试：

- 配置解析逻辑集中在 `src/config.js`，便于测试复用
- 协议服务在 `src/createServers.js` 中集中创建，便于本地随机端口启动
- 关卡路由按模块拆分，便于选择性装配测试应用
- 第 `25`、`26` 关通过可注入存储目录的工厂函数支持隔离测试

### 11.3 自动化与人工边界

当前自动化测试已覆盖：

- 健康检查
- 关键 HTTP 行为
- 第 `15` 关 WebSocket
- 第 `17` 关 HTTPS Trailer
- 第 `25`、`26` 关的关键状态行为

第 `21` 关 HTTP/2 与 `103 Early Hints` 当前保留人工验收边界。原因不是该能力不重要，而是其验证更依赖真实 HTTP/2 客户端行为和更底层的协议细节；在第一阶段内，人工验收比引入高成本、低收益的自动化更符合维护约束。

## 12. 主要架构决策与权衡

### 12.1 采用 Node/Express 单体结构

第一阶段选择 Node/Express 单体结构，而不是拆分多服务。原因是：

- 关卡总量有限
- 第一阶段以单机、低成本维护为前提
- 页面、API、协议和测试装配在同一仓库内更容易同步维护

代价是协议型能力和状态型逻辑会在同一项目中共存，需要文档和模块边界保持清晰。

### 12.2 后端逻辑按关卡拆分

系统没有为所有关卡构建一个高度抽象的“通用谜题引擎”，而是按关卡拆分后端路由。原因是：

- 每关玩法差异大
- 维护者更需要快速定位具体关卡，而不是理解一套过度通用框架
- 小而直观的本地实现更符合当前项目的长期维护策略

### 12.3 协议能力不强行塞进统一代理路径

对于第 `15`、`17`、`21` 关，系统明确承认代理层兼容性存在不确定性，因此允许或要求客户端接触更直接的 Node 协议入口。这一选择优先保证玩法真实可达，而不是追求所有能力都走统一反向代理出口。

### 12.4 状态型关卡使用文件持久化

第 `25`、`26` 关没有引入数据库，而是使用文件系统中的 JSON 状态文件。选择理由是：

- 状态规模小
- 会话隔离边界明确
- JSON 可读、可调试、可人工检查
- 更符合单机和低维护成本前提

代价是：

- 需要显式处理文件缺失、损坏和过期
- 需要用白名单校验和独立目录避免会话串扰

### 12.5 第 21 关保留人工验收边界

当前未把第 `21` 关完整纳入自动化协议测试，不是因为其不重要，而是因为第一阶段更看重“稳定、可执行、维护成本可控”的验证方式。该关卡因此采用 SRS 中显式定义的人工验收步骤，保持测试边界清晰可追踪。
